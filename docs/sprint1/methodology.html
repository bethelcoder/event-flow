Git Methodology - Sprint 1

Methodology Used

We follow the GitHub Flow methodology to manage our codebase efficiently. This ensures collaboration, minimizes conflicts, and keeps the main branch stable at all times.

Branching Strategy

The main branch is locked for direct changes. Every team member works on their own feature or documentation branch. Examples include:

  • documentation-site ‚Äî for documentation site work
  • frontend-feature-login ‚Äî for login page frontend development
  • backend-api-events ‚Äî for API endpoints related to events

Once a branch is ready, a pull request (PR) is created to merge into main.

Pull Requests & Code Review

Each PR is reviewed by at least one team member before merging. This ensures:

  • Code quality and readability
  • Consistency with project standards
  • Detection of potential bugs early

The main branch manager approves the PR if it meets all criteria.

Commit Practices

We follow these commit rules to make history clear and trackable:

  • Write descriptive commit messages (e.g., Added RSVP backend endpoint)
  • Commit frequently to save progress
  • Push branches regularly to GitHub
  • Include task IDs when applicable for easier reference

Additional Notes
